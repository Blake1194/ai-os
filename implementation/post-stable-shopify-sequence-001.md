# Post-Stable Shopify Sequence 001

## Purpose
Define the revenue-oriented next major track after Atlas reaches a stable, workable assistant baseline.

## Strategic Position
The operator wants:
- a stable workable Atlas first
- then Shopify launch help to start producing revenue
- then later refinement/polish of Atlas

That is a good sequence.

## Recommended Order
### Phase A — Stable Assistant Baseline
- interactive local/runtime proof
- voice path beginning
- basic multi-surface control
- visible consent and override model

### Phase B — Shopify Finish And Launch Support
Use Atlas to help finish and launch the Shopify store by focusing on:
- remaining content gaps
- product/page QA
- policy page checks
- navigation/link checks
- launch checklist
- post-launch monitoring tasks
- support/comms templates

### Phase C — Atlas Polish And Expansion
After revenue motion begins:
- improve UX
- improve voice flow
- improve parallel workers
- deepen integrations
- improve interface polish

## Why This Order Makes Sense
- the assistant becomes useful now
- Shopify creates real-world output and possible revenue sooner
- later Atlas polish becomes easier once the system is already proving value

## Recommended Shopify Worker Types Later
- storefront QA worker
- product-page completion worker
- policy/compliance checklist worker
- launch checklist worker
- post-launch monitoring worker

## Exact Near-Term Rule
Do not let Shopify become a distraction from proving the assistant runtime.
But once the assistant baseline is stable, Shopify is an excellent next outcome-focused lane.
