# Intake Workflow

## Purpose
Define how new ideas, screenshots, transcripts, files, and requests enter the system.

## Input Source Types
- screenshots
- transcripts
- voice notes
- browser pages
- GitHub pages
- email content
- project ideas
- tool discoveries
- automation opportunities

## Classification Rules
Classify incoming material as one or more of the following:
- task
- note
- decision
- workflow candidate
- command candidate
- reference only
- ignore or defer

## What Should Become a Task
- anything with a clear desired outcome
- anything requiring action, setup, or follow-up
- anything time-sensitive or operational

## What Should Become a Note
- observations
- context
- raw findings
- partial ideas not ready for execution

## What Should Become a Decision
- choices that affect tools, architecture, priorities, or operating rules

## What Should Become a Workflow
- repeated processes
- multi-step actions that should be standardized
- tasks that need safe checkpoints

## What Should Become a Command Template
- requests that are likely to be reused in similar wording
- task types that benefit from consistent invocation

## What Should Be Ignored or Deferred
- low-value noise
- duplicate discoveries with no new value
- ideas with no current relevance or action path

## Intake Rule
New material should enter the system in the simplest durable form possible:
- task if action is needed
- note if context is needed
- decision if direction changes
- workflow if repetition is likely
- command if reuse is likely
